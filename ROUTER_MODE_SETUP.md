# Router Mode Setup ‚Äî Idle Model Unload Without Killing the Server

> **Goal:** Single model, served behind a cloudflared tunnel, auto-unloaded after 10 minutes of inactivity to save energy. The llama-server process itself stays alive across unloads so the tunnel never breaks.

---

## Current Setup (Before)

```
Cloud Tunnel (cloudflared)
  ‚Üí Port 8001 (proxy.py)
    ‚Üí Port 8002 (llama-server.exe b8996, --model ‚Ä¶)
      ‚Üí Single model loaded at startup
      ‚Üí "Unload" = kill the entire process
      ‚Üí Server restart breaks the tunnel
```

**Problems:**
- No in-process model unload ‚Äî `/models/unload` calls `process.terminate()`.
- Killing llama-server drops the tunnel's upstream socket.
- Old build (b8996, May 1, 2025).

---

## New Setup (After)

```
Cloud Tunnel (cloudflared)
  ‚Üí Port 8001 (proxy.py ‚Äî updated)
    ‚Üí Port 8002 (llama-server.exe b9094, router mode)
      ‚Üí Router process stays alive forever
      ‚Üí Model is a child process, loaded on first request
      ‚Üí Idle 10 min ‚Üí proxy calls /models/unload ‚Üí VRAM freed
      ‚Üí Router still up ‚Üí tunnel still up
```

**Benefits:**
- `POST /models/unload` frees VRAM without killing the server.
- Tunnel survives unloads. No reconnect storms.
- Latest build (b9094, May 10, 2026).

---

## Use case assumed in this document

- One model at a time (Qwen3.6-35B-A3B by default; Q3/Q4/27B variants picked via proxy flags).
- Idle timeout: 10 minutes. After that, the model unloads but the router keeps running.
- First request after an unload pays the model load cost (~seconds), not the server boot cost.
- Energy: with no model loaded, GPU draw drops to idle.

## Step 3 ‚Äî Model & context selection

`models-preset.ini` is **generated by `proxy.py` at startup**, not hand-edited. The choices live in two places:

1. **Available models** are defined as a `MODELS` list in `proxy.py` (one entry per weights+mmproj pair):
   - `Qwen3.6-35B-A3B Q3` ‚Üí `Qwen3.6-35B-A3B-UD-Q3_K_XL.gguf`
   - `Qwen3.6-35B-A3B Q4` ‚Üí `Qwen3.6-35B-A3B-UD-Q4_K_M.gguf`
   - `Qwen3.6-27B Q4` ‚Üí `Qwen3.6-27B-UD-Q4_K_XL.gguf`
2. **Available contexts** are `CTX_CHOICES = [32768, 65536, 131072]` (32k / 64k / 128k).

At launch, the proxy presents two arrow-key menus ‚Äî first the model, then the context ‚Äî and writes the result to `models-preset.ini` as a single section named `[qwen3.6-35b-a3b]`. **The section name is intentionally fixed** so pi (which has `defaultModel: "qwen3.6-35b-a3b"` hardcoded in `~/.pi/agent/settings.json`) keeps working regardless of which weights are chosen.

To skip the menus (headless launch):
```powershell
.\watchdog.ps1 --model "Qwen3.6-35B-A3B Q3" --ctx-size 32768
```

To add a 4th model: append a `ModelChoice(...)` entry to the `MODELS` list in `proxy.py:30`. To add a 4th context size: extend `CTX_CHOICES`. No INI editing.

**Confirmed INI keys (router accepts these):** `model`, `mmproj`, `ctx-size`, `n-gpu-layers`, `flash-attn`, `cache-type-k`, `cache-type-v`, `no-mmap`, `parallel`, `jinja`, `temp`, `top-p`, `top-k`. Router normalizes `temp` ‚Üí `temperature` and `no-mmap = 1` ‚Üí `mmap = false` when echoing the preset back; either form is accepted on input.

**VRAM budget (3090 Ti, 24 GB):**
- Q3 35B: ~16 GB weights + KV. Fits at all three contexts (32k = ~16 GB total, 128k ‚âà 22 GB).
- Q4 35B: ~21 GB weights. Tight even at 32k; 128k will OOM.
- Q4 27B: ~16 GB weights. Fits at all three contexts.
- KV cache uses q4_0 (cheap) regardless of context size.

If a load attempt OOMs, the router's load returns `failed: true`; pick a smaller context next time.

## Step 5 ‚Äî Rewrite `proxy.py` for router mode

**Done.** The new `proxy.py` is in place and verified end-to-end. Before/after summary:

| Aspect | Old | New |
|---|---|---|
| Binary | `llama.cpp_setup\llama-server.exe` | `llama.cpp_latest\llama-server.exe` |
| Server command | `--model ‚Ä¶ --mmproj ‚Ä¶ --alias ‚Ä¶` | `--models-preset ‚Ä¶ --models-max 1 --no-models-autoload` |
| Variant selection | `--q4` / `--27b` / `--long` argparse flags | `--default-model` flag (used when client omits `"model"` field) |
| Model lifecycle | Process start/kill | Per-model `/models/load` and `/models/unload` |
| Idle action | `process.terminate()` (kills server, breaks tunnel) | `POST /models/unload` (router stays up) |
| File size | ~1070 lines | ~470 lines |

Architecture:

- **Two lifecycles.** The router process is owned by `lifecycle_context`: spawned on proxy startup, killed on proxy shutdown. The model lifecycle is owned by `ModelManager`: loaded on first request, unloaded by the watchdog. The watchdog never touches the router process.
- **`ModelManager.ensure_loaded(model)`** POSTs `/models/load`, then polls `/v1/models` (every 0.5s, up to `LOAD_TIMEOUT=300s`) until `status.value == "loaded"`. Serialized via an `asyncio.Lock` so two concurrent requests can't double-load.
- **`ModelManager.unload(reason)`** POSTs `/models/unload` for whatever's currently loaded. Idempotent ‚Äî safe to call when nothing is loaded.
- **Idle watchdog** runs every `IDLE_CHECK_INTERVAL=2s`, calls `unload_if_idle()` which checks `_active == 0` and `time - _last_activity >= IDLE_TIMEOUT`. **`IDLE_TIMEOUT` is currently `10` for testing ‚Äî bump to `600` (10 minutes) for production.**
- **Per-request model selection.** The proxy reads the `"model"` field from the request body for `/v1/chat/completions`, `/v1/completions`, `/v1/embeddings`. Falls back to `--default-model` (default `qwen3.6-35b-a3b`) when missing.
- **Adopted-server logic dropped.** The old `_check_existing_server` reconnect branch is gone ‚Äî with a long-lived router it added complexity without value.
- **TUI picker** lives in `pick_setup()` / `_arrow_picker()`. Stdlib only (`msvcrt` on Windows, ANSI VT enabled via `os.system("")`). Two sequential menus: first the model from the `MODELS` list, then the context from `CTX_CHOICES`. After the second menu, `write_preset()` regenerates `models-preset.ini` with one section using the chosen weights + ctx. `--model` and `--ctx-size` skip the respective menus; passing both gives a fully non-interactive launch. No TTY ‚Üí falls back to the first option of each list with a warning. `q`/Esc/Ctrl-C cancels.

The proxy's externally-visible API (`/v1/chat/completions`, `/v1/models`, `/health`, etc.) is unchanged ‚Äî clients behind the tunnel don't see the migration.

`watchdog.ps1` is a thin restart-on-crash supervisor (~30 lines). It calls `.venv\Scripts\python.exe proxy.py` and forwards every flag through. Crash loops are throttled (5 fast restarts ‚Üí exponential backoff up to 60 s).

You can also call `proxy.py` directly without the supervisor:

```powershell
.\.venv\Scripts\python.exe proxy.py [--default-model ‚Ä¶] [--idle-timeout ‚Ä¶] [--proxy-port ‚Ä¶]
```

The old per-variant scripts (`launch_qwen.ps1`, `launch_qwen_q4.ps1`, `launch_27b.ps1`, `watchdog_qwen.ps1`, `watchdog_27b.ps1`) are gone ‚Äî variant is a per-request decision now.

Verified behavior on this machine:

1. Proxy spawns the router on 8002 with `--no-models-autoload`. `/health` reports `loaded: null`, GPU idle.
2. First chat request to `/v1/chat/completions` triggers `ensure_loaded` ‚Üí POST `/models/load` ‚Üí polls `/v1/models` until `loaded` ‚Üí forwards.
3. Cold-load latency for 35B-A3B Q3 at 32k ctx ‚âà 15‚Äì20s on a 3090 Ti. After that, requests pass through directly.
4. After **10 seconds** of idle (`IDLE_TIMEOUT=10`, the testing value), watchdog POSTs `/models/unload`. VRAM frees. Router process still alive (`Get-Process llama-server`).
5. Next request ‚Üí reload, same flow. Confirmed working.
6. `Ctrl+C` on the proxy ‚Üí router process is terminated cleanly via `lifecycle_context`.

To bring it to production: change `IDLE_TIMEOUT = 10` ‚Üí `600` (10 minutes) in `proxy.py`, or pass `--idle-timeout 600`.

### Cloudflared compatibility

`H:\cloudflared\` is unchanged. Its `~/.cloudflared/config.yml` ingress rule is:

```yaml

  - hostname: ai.htk-hrt.cc
    service: http://localhost:8001
```

What this migration delivers for the tunnel: when the watchdog unloads the model, the router process keeps the TCP socket alive on `:8002`, the proxy keeps `:8001` listening, and cloudflared's QUIC connections to Cloudflare edge stay intact. Previously, model unload meant `process.terminate()` on llama-server, which broke `localhost:8002` ‚Üí tunnel-side 502s on the next request. That failure path is gone.
